# Дневник выполнения лабораторной работы №1

## 1. Настройка окружения

Подняла PostgreSQL в Docker Compose. Использовала образ postgres:15, порт 5432 пробросила наружу, настроила сохранение данных. В контейнер также закинула папку с CSV и SQL-файлы, чтобы база сразу создавалась с нужной структурой.

## 2. Импорт данных

Загружала CSV-файлы через DBeaver. Первый файл встал без проблем, но при добавлении второго выскочила ошибка — id повторяются, потому что в каждом файле нумерация идёт заново. Решила импортировать все 10 файлов по очереди в режиме Insert, а не Replace, чтобы данные добавлялись, а не затирались. В итоге в mock_data получилось 10000 строк.

## 3. Анализ сырых данных

Рассмотрела таблицу mock_data. В ней 56 колонок, каждая запись — отдельная продажа. Информация о покупателе, продавце, товаре, магазине и поставщике свалена в одну строку.

Проверила через SQL:
- Посмотрела первые 10 записей, чтобы понять структуру
- Выяснила, что email покупателя и продавца уникальны, а названия товаров часто повторяются
- Насчитала 383 уникальных магазина, 383 поставщика, 3 категории товаров (Cage, Food, Toy) и 5 типов питомцев (Dogs, Cats, Birds, Fish, Reptiles)

## 4. Проектирование схемы

На основе анализа придумала нормализованную схему "снежинка". Получилось 15 таблиц: одна фактовая и семь измерений. В "снежинке" измерения дополнительно дробятся — например, категория товаров вынесена в отдельную таблицу, в отличие от "звезды", где всё хранится в одной таблице товаров.

## 5. Создание и заполнение таблиц

Написала DDL-скрипт: сначала создаются независимые таблицы (категории), потом те, которые на них ссылаются (товары), затем остальные измерения, и в самом конце — таблица фактов.

Написала DML-скрипт для переноса данных. Использовала INSERT с подзапросами и JOIN, чтобы из mock_data вытащить нужные поля и связать с id из таблиц измерений. Например, для fact_sales собирала id покупателя по email, id товара по названию и бренду, id магазина по названию и так далее.

Была проблема: source_customer_id дублируется, потому что в разных файлах id начинаются с 1. Убрала уникальность для этого поля, оставив уникальность только по email. После этого скрипты заработали.

## 6. Проверка результата

После запуска контейнера проверила, что в fact_sales 10000 записей — столько же, сколько в исходной mock_data.

Сравнила суммы продаж: общая выручка в mock_data и в fact_sales совпала. Значит, данные перенеслись целиком, ничего не потерялось.

Проверила измерения на дубликаты. Например, для покупателей:

```sql
SELECT customer_email, COUNT(*) FROM analytics.dim_customer GROUP BY customer_email HAVING COUNT(*) > 1;



Пустой результат — дубликатов нет. Аналогично проверила продавцов, магазины и поставщиков.

Проверила внешние ключи, чтобы не было "висячих" записей в фактах:


```sql
SELECT COUNT(*) FROM analytics.fact_sales f 
LEFT JOIN analytics.dim_customer c ON f.customer_id = c.customer_id 
WHERE c.customer_id IS NULL;

Вернулся 0 — все связи корректны.

Проверила, что категории действительно вынесены отдельно:

```sql
SELECT p.product_name, c.category_name 
FROM analytics.dim_product p
JOIN analytics.dim_category c ON p.category_id = c.category_id
LIMIT 5;

Запрос вернул товары с их категориями — всё работает как надо.
Все проверки пройдены. Схема "снежинка" построена.
